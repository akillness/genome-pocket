# 02 · 타진 가능한 업무 범위 (Workable Scope)

> pm-skills의 *Opportunity Solution Tree(Teresa Torres)* 와 *RICE 우선순위*로 범위를 구조화한다.
> 목표는 "다 만든다"가 아니라 "어디까지가 3시간, 어디부터가 다음 단계인지"를 명확히 가르는 것.

## 1. Opportunity Solution Tree

```
[Desired Outcome]

 (North Star: 자동 분류·아카이빙된 유효 보도 건수/월)
        │
   ┌────┼─────────────────────┬───────────────────────┐
   │    │                     │                       │
[Opp.1]            [Opp.2]                 [Opp.3]              [Opp.4]

 ─수신 뉴스를 저장소에  ─사업계획서 키워드     ─LLM 정성평가:          ─기사에서 언론사/기자
```

Opp.3(정성평가)이 "보도 횟수를 넘어선" 차별적 가치지만 가장 불확실 → 작은 실험으로 신호만 확인.

## 3. 범위 티어 (무엇을 약속하고 무엇을 보류할지)

### Tier 0 — Must (3시간 안에 반드시 흐르게)
- 샘플 뉴스(잔디 export 또는 알리미)를 **하나의 저장소에 적재** + 중복 제거 → "휘발" 문제 즉시 해소.
- 키워드 오탐 1차 필터.
- 활동가가 보는 **얇은 대시보드/시트 뷰** 1개(날짜·제목·언론사·링크·자동 태그 컬럼).

### Tier 1 — Should (시간이 남으면)
- 사업/팀 **자동 분류** 태깅(규칙 + LLM 보정), 정확도 즉석 측정.
- 언론사/기자 추출 컬럼 추가.

### Tier 2 — Could / 실험
- 보도자료 복사 vs 심층 기사 **LLM 정성평가** PoC: 라벨 6건으로 "판별 가능성"만 확인하고 정확도를 솔직히 보고.

### Tier 3 — Won't (이번엔 안 함, 백로그)
- 실시간 자동 수집 파이프라인 운영화(스케줄러·모니터링).
- 본격 미디어 통계 대시보드(추이·기자 네트워크).
- 권한·멀티유저·배포 자동화.

→ "Won't"을 명시하는 이유: 활동가/사무처장과 **기대치를 정렬**하고, 다음 협업의 백로그를 남기기 위함.

## 4. 의존성·전제 (이게 없으면 범위가 막힌다)

- 잔디/알리미 데이터에 **언론사·날짜·URL 메타**가 남아야 S4가 가능. 없으면 본문 크롤링 필요(Effort↑, 저작권 확인 필요).
- 사업/팀 분류 기준이 **문서로 존재**해야 S3가 규칙 기반으로 빠르게 출발. 없으면 LLM zero-shot 태깅으로 시작.
- 본문 텍스트 확보 + 외부 LLM 전송 허용이 있어야 S5(정성평가) 실험 가능.

## 5. 개발자 강점과의 정렬

매칭 메모상 강점(멀티모달 RAG, API 서버, Slack/Bot 에이전트)은 **S3·S5(분류·정성평가)** 와 직접 맞닿는다.
다만 워크숍 인수인계 관점에서는 화려한 파이프라인보다 **단체가 유지 가능한 표면(구글 생태계)** 위에 강점을 얹는 것이 적합하다.
