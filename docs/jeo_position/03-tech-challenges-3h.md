# 03 · 3시간 기술 도전과제 (3-Hour Technical Challenges)

> pm-skills의 *PRD / Spec / Sprint* 관점. 3시간을 "얇은 수직 슬라이스"로 끝까지 흐르게 만든다.
> 원칙: **데이터 적재(휘발 해결)부터 → 분류 → 정성평가 실험** 순으로, 각 단계가 독립적으로 데모 가능하도록.

## 0. 성공 정의 (Definition of Done, 워크숍 종료 시)

- [ ] 샘플 뉴스가 **하나의 영구 저장소**에 적재되고 새로고침해도 사라지지 않는다. (휘발 문제 해소 = 핵심 성과)
- [ ] 활동가가 **버튼/링크 한 번**으로 결과(목록 + 자동 태그)를 볼 수 있다.
- [ ] "수집 → 분류 → 대시보드 카드 1줄"이 **end-to-end로 1건이라도 흐른다.**
- [ ] 정성평가는 가능성/한계를 **솔직한 수치로** 보고한다(과대약속 금지).
- [ ] 활동가가 키워드 추가/키 갱신을 할 수 있는 1쪽 운영 메모를 남긴다.

## 1. 시간대별 플랜 (3시간)

| 시간 | 활동 | 산출물 |
|------|------|--------|
| 0:00–0:25 | (사전 25분 싱크 별도) 데이터 샘플 함께 열람, MVP 시나리오 확정 | MVP 합의 |
| 0:25–1:10 | **Tier 0**: 샘플 적재 + 정규화 + 중복 제거 + 키워드 오탐 필터 | 깨끗한 데이터 테이블 |
| 1:10–1:20 | 활동가와 중간 점검(이 목록이 맞나?) | 피드백 1회 |
| 1:20–2:10 | **Tier 1**: 사업/팀 자동 분류(규칙 + LLM 태깅) + 대시보드 뷰 | 태깅된 대시보드 |
| 2:10–2:40 | **Tier 2 실험**: 보도자료 복사 vs 심층 LLM 판별 PoC, 정확도 측정 | 정성평가 신호 + 수치 |
| 2:40–3:00 | 데모 + 운영 메모 + 다음 단계 백로그 정리 | 인수인계 패키지 |

> 시간이 부족하면 Tier 1 분류를 줄이고 **Tier 0 + 대시보드**를 반드시 완성한다(데이터 휘발 해소가 단일 최대 가치).

## 2. 도전과제별 기술 노트

### Challenge A — 데이터 적재 & 정제 (Tier 0, 핵심)
- **문제**: 잔디로만 흘러가서 기록이 휘발. 키워드 오탐(동음이의·무관 기사).
- **접근**:
  - 입력 소스: ① 잔디 export(CSV/JSON) ② 구글 알리미 RSS(`https://www.google.com/alerts/feeds/...`) ③ (필요시) 시트.
  - 적재 대상: **구글 시트**(인수인계 최적) 또는 경량 DB(SQLite/로컬). 단체 유지보수성을 위해 시트 1순위.
  - 중복 제거: URL 정규화(추적 파라미터 제거) + 제목 유사도(정규화 후 해시/임베딩).
  - 오탐 필터: 규칙(필수 키워드 AND/제외어 NOT) → 경계 사례만 LLM 분류로 보정.
- **이전 실패 교훈 반영**: 앱스스크립트에서 막힌 "기간 설정·키워드 정확도"는 ① 명시적 날짜 윈도우 파라미터 ② 규칙+LLM 2단 필터로 해소.

### Challenge B — 사업/팀 자동 분류 (Tier 1)
- **문제**: 기사를 담당 팀·세부 사업으로 분류하는 게 수작업.
- **접근**:
  - 분류 기준이 문서로 있으면 → 키워드 매핑 테이블(사업명 ↔ 키워드)로 1차 규칙 분류.
  - 모호/미매칭 건 → LLM few-shot 태깅(사업 목록을 컨텍스트로 주고 multi-label).
  - **정확도 즉석 측정**: 활동가가 20건 라벨링 → 자동 태그와 대조 → 정확도 표로 제시.
- 강점(RAG) 활용: 사업계획서를 청크·임베딩해 기사와 의미 유사도로 매칭하는 미니 RAG도 가능(시간 여유 시).

### Challenge C — 보도 깊이 정성평가 PoC (Tier 2, 실험)
- **문제**: "보도자료 복사 기사" vs "활동가 코멘트·기자 추가 취재 심층 기사" 구분 = 차별적 가치이자 가장 불확실.
- **접근**:
  - 신호 정의: 단체 코멘트/인용 존재, 기자 바이라인·추가 취재 정황, 보도자료 원문과의 텍스트 중복률.
  - LLM 분류 프롬프트(라벨 6건 = few-shot) → "복사/혼합/심층" 3분류 + 근거 + 신뢰도.
  - (옵션) 보도자료 원문 대비 n-gram/임베딩 유사도로 "복사" 정량 보강.
  - **솔직한 보고**: 6건은 통계가 아니다. "이 신호로 판별 가능해 보인다/추가 라벨 N건 필요" 수준으로 결론.

### Challenge D — 언론사·기자 아카이빙 (Tier 1 스트레치)
- 본문/메타에서 언론사·기자명 추출(규칙 + LLM NER) → 시트 컬럼 → 언론사별 보도 수 집계.
- 미디어 소통 통계의 씨앗(다음 단계에서 추이·랭킹으로 확장).

### Challenge E — 접근성 대시보드 (Tier 0 표면)
- 후보: ① 구글 시트 + 필터/피벗(가장 낮은 문턱) ② Looker Studio(시트 연결, 차트) ③ Streamlit(개발자 친화·배포 부담).
- 활동가 기준(앱스스크립트 메뉴 버튼 수준) → **시트/Looker Studio 1순위**. 버튼=Apps Script 메뉴로 "갱신" 트리거.

## 3. 기술 스택 후보 비교

| 표면 | 문턱(활동가) | 구축 속도 | 인수인계 | 비고 |
|------|:---:|:---:|:---:|------|
| 구글 시트 + Apps Script 메뉴 | ◎ | ◎ | ◎ | 단체가 이미 구글 사용, 버튼 수준 OK |
| Looker Studio (시트 소스) | ◎ | ○ | ◎ | 차트·필터 무료, 코드 적음 |
| Streamlit / 경량 웹 | △ | ○ | △ | 개발자 강점, 배포·유지 부담 |
| 백엔드 API + DB | △ | △ | △ | 운영화 단계용, 워크숍엔 과함 |

권장: **수집·분류 로직은 Python/Apps Script**, **저장·표시는 구글 시트 + Looker Studio**. 강점인 LLM/RAG는 분류·정성평가 모듈에 국한.

## 4. 리스크 & 대응

| 리스크 | 영향 | 대응 |
|------|------|------|
| 데이터 메타 부족(언론사·기자 없음) | C/D 막힘 | Tier 0/B로 범위 축소, 본문 크롤링은 저작권 확인 후 다음 단계 |
| 분류 기준 문서 부재 | B 지연 | LLM zero-shot 태깅으로 시작, 기준은 워크숍에서 함께 초안화 |
| 외부 LLM 전송 제약 | C 불가 | 본문 미전송·메타만 사용 or 로컬 모델, 가능 범위로 PoC |
| 키워드 오탐 과다 | 신뢰도↓ | 제외어 + 경계 사례 LLM 보정, 활동가와 오탐 룰 합의 |
| 시간 부족 | 데모 실패 | Tier 0 + 대시보드를 우선 완성(휘발 해소 단일 최대 가치) |

## 5. 워크숍 후 남길 백로그 (다음 협업)

- 실시간 자동 수집 운영화(스케줄러 + 실패 알림).
- 정성평가 라벨 확장 + 평가 정확도 모니터링.
- 미디어 소통 통계 대시보드(언론사·기자 추이, 의제별 노출).
- 사업계획서 ↔ 기사 RAG 매칭 고도화.
